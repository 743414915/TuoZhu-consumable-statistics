## Android 耗材管理 MVP 测试方案

### 测试范围
- 界面层：主界面耗材列表、添加耗材、编辑耗材、滤器和搜索功能、设置提醒（如存在）。
- 业务层：耗材剩余克重的录入/更新、耗材使用记录计数、同步本地数据到持久层。
- 数据层：Room/本地数据库的增删改查逻辑、数据迁移、数据一致性。
- 综合流程：从新增耗材、更新剩余克重、标记为用完、再添加下一卷的完整生命周期。

### 风险点
1. 设备无 AMS，耗材重量需要用户手动估算，易导致输入错误；需要验证界面提示与输入校验（范围、单位）。
2. 多线程或异步写入 Room 可能造成数据丢失或 UI 回显异常，需要验证事务与 LiveData/Flow。
3. 没有云服务依赖，用户卸载/重装后数据保持是主要风险，需验证备份或导出机制（如有）。
4. 多卷耗材共存时的排序、区分逻辑（例如未打包与已使用）必须明确。
5. 在网络中断情况下，含有同步或推送机制的模块需保障不影响本地流程。

### 手工回归清单
1. 新建耗材，填写类型、剩余克重、时间、备注，确认列表内新增条目并可编辑。
2. 编辑已有耗材克重，检查界面回显与本地存储一致。
3. 标记耗材为“用完”并添加下一卷，确认状态切换正确、排序更新。
4. 使用搜索/过滤条件（如类型、状态）查询耗材，检查结果准确无残留。
5. 重启应用、切换横竖屏（如适用）后数据仍可用。
6. 异常输入（空、负数、过大）触发校验提示，不能保存。
7. 多条耗材并发更新（模拟批量操作），确保 UI 不会崩溃。
8. 新建耗材时，提醒/通知机制（如果存在）按预计时间/剩余量触发。
9. 桌面同步草稿重复上报同一 `externalJobId` 时，只保留一条待确认任务，并允许在 `DRAFT` 状态下刷新内容。
10. 桌面同步草稿包含非法材料、非法时间戳、非正克重时，只忽略坏数据并保留有效数据。
11. 用户确认待确认打印任务后，只生成一条 `PRINT_USAGE` 事件；再次确认同一任务不能重复扣减。
12. 没有活动卷时，确认待确认打印任务必须被拦截并提示用户先设置活动卷。

### 回归执行策略（至少 5 轮）
1. 初始冒烟轮：功能是否能启动并完成“添加 + 编辑 + 标记用完”的核心路径，重点观察崩溃与明显数据异常。
2. 功能回归轮：全量执行手工回归清单，记录每条用例状态。
3. 稳定性轮：聚焦并发操作、数据库写入与 UI 反馈之间的协同，重点探查 race condition。
4. 兼容性轮：在不同安卓版本/分辨率模拟器（或真机，如 Android 11/12/13）上执行清单。
5. 验收前最终轮：模拟真实使用（连续 5 次新增/耗尽/添加新卷）并确认所有数据保持一致后冻结版本。
每轮都至少由 2 名工程师一人执行用例、一人审核结果并确认问题是否回归。

### 验收口径
- 核心流程（添加→编辑→标记完成）无崩溃、数据一致，所有校验提示明确。
- 数据持久化成功，重启后列表完整。
- 所有高风险用例（输入校验、并发更新、状态切换）均在回归轮中通过。
- 交付时附带回归记录（通过/失败/重测次数）和未解决问题的优先级。

### 后续自动化建议
1. 通过 Espresso/Compose Test 书写界面层自动化用例，覆盖新增、编辑、状态变化流程。
2. 利用 Robolectric 或 Instrumentation 进行 Room DAO 的逻辑测试，确保事务和数据一致性。
3. 设立数据回归脚本（如基于 Room 实例）定期校验历史耗材数据格式与完整性，配合 CI 提示异常。
4. 增加持续回归的测试套件，在核心功能稳定后逐步纳入夜间构建。
5. 为桌面同步器增加独立的 PowerShell 回归脚本，覆盖样例拉取、确认回执、重复上报和坏数据告警。
